# 代理机制：系统代理、TUN 与 DNS 分流

这一篇看日常代理工具是怎么接管流量的：系统代理只是应用层约定，TUN 模式从网络层接管，DNS 分流决定域名解析走哪条路径。

## 4. 系统代理的工作原理

### 4.1 什么是系统代理

**系统代理**（System Proxy）是操作系统提供的一个设置项——它本质上是一个**约定**，告诉所有「遵守约定」的应用程序：「把你的网络请求发给这个地址」。

它不是操作系统强制拦截流量——更像是一个**告示牌**，上面写着「代理在 127.0.0.1:7890」，程序路过时可以读，也可以无视。

```text
应用程序发起网络请求
    │
    ├─ 遵守系统代理？ ──→ 读取系统代理地址 → 发给代理
    │
    └─ 不遵守？ ──→ 直接连接目标（绕过代理）
```

**各平台的系统代理设置位置：**

| 平台 | 设置位置 | 本质 |
|------|----------|------|
| Windows | 设置 → 网络和 Internet → 代理 | 注册表 + WinINET/WinHTTP API |
| macOS | 系统偏好设置 → 网络 → 高级 → 代理 | SystemConfiguration 框架 |
| Linux (GNOME) | 设置 → 网络 → 代理 | gsettings / dconf |
| Linux (KDE) | 系统设置 → 网络 → 代理 | KDE 配置框架 |

应用程序通过操作系统提供的 API 读取这些设置——如果它选择了读取的话。

---

### 4.2 三种工作模式

现代代理客户端（Clash Verge、V2RayN、Surge）通常提供三种模式：

#### 全局模式（Global）

所有能匹配到的流量都走代理。实现方式：**接管系统代理设置，将所有请求导向代理**。

优点：简单，不需要维护规则。
缺点：
- 国内网站走代理，延迟增加
- 代理带宽被内网流量占用
- 部分国内服务（银行、政务）检测到海外 IP 会拒绝服务
- 代理服务器的 IP 可能被国内 CDN 限速

#### 规则模式（Rule）

根据预定义的规则集，**按域名或 IP 分流**——这是最推荐的日常使用模式：

```text
流量进入代理客户端
    │
    ├─ 匹配「直连规则」（*.cn、内网 IP 段） → 直接连接
    ├─ 匹配「代理规则」（google.com、twitter.com） → 走代理
    ├─ 匹配「拒绝规则」（广告/跟踪域名） → 直接拒绝
    └─ 未匹配 → 按默认策略（通常是代理）
```

**规则的常见类型：**

| 规则类型 | 示例 | 说明 |
|----------|------|------|
| 域名后缀 | `DOMAIN-SUFFIX,google.com,Proxy` | google.com 及其子域名走代理 |
| 域名关键词 | `DOMAIN-KEYWORD,google,Proxy` | 域名含 "google" 的走代理 |
| IP CIDR | `IP-CIDR,10.0.0.0/8,DIRECT` | 内网 IP 段直连 |
| GEOIP | `GEOIP,CN,DIRECT` | 根据 IP 地理位置（中国 IP 直连） |
| 进程名 | `PROCESS-NAME,Telegram,Proxy` | 指定进程走代理 |

#### 直连模式（Direct）

所有流量都不走代理——相当于**暂时关闭代理功能**。代理客户端仍在运行，但不介入任何流量。

> [!NOTE]
> 规则模式 + 全局模式 + 直连模式 = 决策的是「什么流量走代理」；这与透明代理/显式代理（决策的是「流量怎么进入代理」）是正交维度。你可以在规则模式下使用显式代理（配置系统代理），也可以在规则模式下使用透明代理（TUN 接管 + 规则分流）。

---

### 4.3 系统代理的局限性

系统代理是**应用层代理**——只有遵守约定的应用才会走代理：

| 类型 | 走代理？ | 说明 |
|------|----------|------|
| 浏览器（Chrome、Firefox、Edge） | ✅ | 原生支持系统代理 |
| HTTP 客户端（curl、wget、Python requests） | ⚠️ | curl 可以读系统代理（取决于编译选项），大多数 CLI 工具不行 |
| 终端（大部分 CLI） | ❌ | git、npm、pip 依赖环境变量，不读系统代理 |
| 桌面应用 | ❌ | 大部分不查系统代理设置，直接连接 |
| ping、dig、nslookup | ❌ | 不经过 HTTP 协议层 |
| UDP 流量 | ❌ | 大部分系统代理实现只代理 TCP |

这就是为什么需要 **环境变量代理**（`HTTP_PROXY`）和 **TUN 模式**（§5）来覆盖系统代理覆盖不到的场景。

## 5. TUN 模式

### 5.1 什么是 TUN 模式

TUN 模式通过创建**虚拟网卡**，在操作系统网络栈的网络层（OSI 第 3 层）拦截流量——不再依赖应用程序是否「遵守代理约定」，而是**从 IP 包层面截获所有出站流量**。

```text
应用程序（任意程序——浏览器、游戏、终端工具）
    │
    ▼
操作系统网络栈
    │
    ▼
TUN 虚拟网卡 ←── 代理客户端用户态程序处理
    │
    ├─ 直连流量 → 物理网卡 → Internet（直接连接）
    └─ 代理流量 → 封装 → 代理服务器 → 目标
```

**和系统代理的本质区别：**

- 系统代理：应用程序**主动**问「代理在哪」，然后发给代理 → 应用层约定
- TUN 模式：操作系统**被迫**把 IP 包发给 TUN 网卡，应用程序不知道 → 网络层劫持

> TUN 模式在效果上等于「透明代理 + 分流规则」——流量被透明地（客户端无感知）截取到代理客户端，然后在内部做分流。

---

### 5.2 工作原理（逐步）

1. **创建虚拟网卡**：代理客户端调用操作系统 API，创建 TUN 设备（Linux 上为 `/dev/tun`，创建后出现 `tun0` 网卡）
2. **修改路由表**：添加路由规则，让所有出站流量（或匹配条件的流量）走 TUN 网卡而不是物理网卡
3. **流量进入 TUN**：内核把 IP 包发给 TUN 设备，TUN 设备另一端连着代理客户端的用户态进程
4. **用户态处理**：代理进程读取原始 IP 包 → 解析源/目标 IP 和端口 → 根据分流规则决定：
   - **直连**：通过原始 socket（`IPPROTO_RAW` 或 `SO_MARK` 绕过 TUN 路由）直接发出
   - **代理**：从 IP 包中提取 TCP/UDP 数据，封装到代理协议（Shadowsocks、VMess 等），发给远程代理服务器
   - **拦截**：丢弃（如广告域名的流量）
5. **返回路径**：远程代理返回的响应 → 代理进程解封 → 写回 TUN 网卡 → 内核交付给应用程序

```text
步骤可视化：

应用发 IP 包
    │
    ▼
路由表：默认网关 → TUN  ←── 关键：路由表被修改了
    │
    ▼
TUN 虚拟网卡（内核侧）
    │
    ▼
代理客户端（用户态进程）
    ├── 解析 IP 包 → 知道目标是谁
    ├── 查分流规则
    ├── 需要代理：封装 → 发往远程代理服务器
    ├── 直连：原始 socket → 物理网卡 → 目标
    └── 拦截：丢弃
```

---

### 5.3 TUN 模式 vs 系统代理

| 维度 | 系统代理 | TUN 模式 |
|------|----------|----------|
| 工作层级 | 应用层（第 7 层） | 网络层（第 3 层） |
| 生效范围 | 遵守约定的应用 | **所有应用**（系统级） |
| TCP 支持 | ✅ | ✅ |
| UDP 支持 | 依赖实现 | ✅ |
| ping / ICMP | ❌ | ✅（可代理 ICMP） |
| 终端工具 | 需手动设环境变量 | 自动生效 |
| 游戏/非标准应用 | ❌ | ✅ |
| 性能开销 | 低（应用程序直接发代理） | 中（用户态处理每个 IP 包，有内核↔用户态切换开销） |
| 配置复杂度 | 低 | 中高（需管理员权限创建虚拟网卡、改路由表） |

> [!WARNING]
> TUN 模式需要管理员/root 权限。配置不当可能导致**所有流量进入 TUN 形成环路**（包括流向代理服务器本身的流量），表现为断网。排查方法：确保到代理服务器 IP 的路由不经过 TUN 网卡。

## 6. DNS 与代理分流

DNS 是代理配置中最容易忽视但又最容易出问题的一环——它能导致「代理连通了但网页打不开/特别慢」这种诡异症状。

### 6.1 为什么 DNS 需要分流

假设你的代理服务器在美国。如果你所有 DNS 查询都通过代理发出：

```text
1. google.com → 代理 DNS 解析 → 返回正确的 IP ✅
2. baidu.com → 代理 DNS 解析 → 返回 baidu 海外 CDN IP → 访问速度极慢 ❌
```

这就是 DNS 不分流的后果——国内域名被解析到海外 CDN 节点，速度从毫秒级退化到秒级。

还有一个隐私问题：**DNS 泄漏**。你本应直连的国内域名，DNS 查询却发到了海外代理，暴露了你正在访问哪些网站。

### 6.2 DNS 分流策略

```text
DNS 查询到达代理客户端
    │
    ├─ 域名匹配「代理规则」 → 通过代理服务器发出 DNS 查询（防 DNS 污染）
    ├─ 域名匹配「直连规则」 → 使用本地 DNS（如 223.5.5.5），获取国内最优 CDN
    └─ 未匹配 → 默认策略（通常是并发查询取最快响应，或走代理 DNS）
```

### 6.3 Fake IP 模式

Fake IP 是 Clash 等现代代理工具的核心 DNS 技术，解决了一个棘手问题：**IP 层面很难做规则匹配**。

**问题场景**：很多服务使用 CDN（内容分发网络），一个域名对应成百上千个 IP，而且 IP 动态变化。如果代理客户端在 IP 层做规则匹配（判断这个 IP 是国内还是国外），几乎不可能准确。

**Fake IP 解决方案**：

1. 应用程序发起 DNS 查询（如 `www.google.com`）
2. 代理客户端**不查询真实 DNS**，而是立即返回一个假 IP（如 `198.18.0.1`，来自保留地址段）
3. 代理客户端在内部维护映射表：`198.18.0.1 → www.google.com`
4. 应用程序用假 IP 发起 TCP 连接，流量到达代理客户端
5. 代理客户端通过假 IP 反查出真实域名 → 根据域名匹配分流规则 → 决定走代理还是直连
6. 走到代理时，由远程代理服务器做真实的 DNS 解析

**Fake IP 的优势**：
- 规则匹配基于域名（准确），而不是 IP（不可靠）
- 所有 DNS 查询瞬间返回（假 IP），延迟极低
- 完美解决 CDN IP 判断困难的问题

**Fake IP 的代价**：
- 直连流量需要代理客户端自行 DNS 解析真实 IP，增加了一跳
- 假 IP 在代理客户端关闭后就失效

> 与透明代理的关系：Fake IP 在 TUN 模式下尤为有用——TUN 模式截获的是 IP 包，如果不做 Fake IP，代理客户端需要在 IP 层面判断归属，极容易出错。Fake IP 让 TUN 模式下的规则匹配回到了域名层面。
