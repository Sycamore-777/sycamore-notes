# 代理概念：方向、协议与边界

这一篇只处理概念边界：代理是什么，正向、反向、透明代理怎么分，HTTP 代理和 SOCKS5 代理有什么差别，以及代理、VPN、路由、NAT 为什么不是一回事。

[返回代理指南入口](proxy-comprehensive.md)

## 1. 什么是代理

### 1.1 核心定义

**代理（Proxy）** 是一台位于客户端和目标服务器之间的中间服务器。关键特征：客户端和目标服务器之间**不建立直接 TCP 连接**，而是建立两段独立连接：

```text
客户端 ── TCP 连接 1 ──→ 代理服务器 ── TCP 连接 2 ──→ 目标服务器
```

请求方向：客户端 → 代理 → 目标
响应方向：目标 → 代理 → 客户端（代理收到响应后，通过连接 1 传回客户端）

```text
客户端 ────→ 代理服务器 ────→ 目标服务器
             ↑                │
             └── 响应返回 ────┘
```

代理在这个模型中的角色不是「转发路由器」，而是**应用层中间人**：

- 客户端知道代理的存在（主动配置了代理地址）；
- 目标服务器看到的请求来源是代理的 IP，而不是客户端的真实 IP；
- 代理**终结了客户端发来的 TCP 连接**，再向目标发起一条**新的 TCP 连接**——这是代理和 NAT 的本质区别（详见 §7.2）。

> 可以把它理解为**收发室**：你不住在目标城市，委托当地收发室帮你寄信和收信。对方看到的寄件地址是收发室的地址，不是你家的地址。
>
> 你 = 客户端，收发室 = 代理服务器，收信人 = 目标服务器
>
> [!TIP]
> 这个类比局限在：收发室只是「转交」，而实际代理还可以做缓存、过滤、修改请求内容、更换协议版本等操作。收发室不能拆开你的信修改内容，但 HTTP 代理可以（对 HTTP 明文而言）。

**代理不是什么**（常见误解）：

- ❌ 代理不是 VPN——代理按应用/域名粒度工作，VPN 接管整个设备网络层（§7.1）
- ❌ 代理不是 NAT——NAT 只改 IP 包头不终结 TCP 连接，代理终结 TCP 并新建连接（§7.2）
- ❌ 代理不一定加密——HTTP 代理可以明文传输；加密取决于代理协议本身是否内置加密层（如 Shadowsocks 加密，HTTP 代理不加密）

### 1.2 为什么代理能访问客户端无法访问的网络

这是很多人第一次接触代理时的核心困惑：**为什么同一个网站，我直接访问不了，但代理能访问？**

答案在于**网络路径不同**：

```text
你的电脑 ──×── 目标服务器        （路径被阻断）
你的电脑 ──✓── 代理服务器 ──✓── 目标服务器  （路径畅通）
```

阻断通常发生在**你到目标服务器之间的某一段路径**上，而不是发生在代理到目标服务器之间。代理服务器位于不同的网络位置（通常是不同的国家或网络运营商），它到目标服务器的路径是完全不同的。

具体原因包括：

| 阻断类型 | 机制 | 代理如何应对 |
|----------|------|-------------|
| **IP 层面阻断** | 你的 IP 段被目标防火墙/ACL 拦截 | 代理 IP 不在拦截范围内 |
| **DNS 污染/劫持** | 本地 DNS 被篡改，解析到错误 IP | 代理使用自己的 DNS，解析到正确 IP |
| **DPI（深度包检测）** | 网络出口设备识别并阻断特定协议（如检测 TLS SNI 字段） | 客户端↔代理之间加密，DPI 看不到 SNI |
| **地理封锁** | 目标服务器按 IP 地理位置拒绝 | 代理位于允许的地理区域内 |

> [!IMPORTANT]
> 关键结论：代理不是「绕过」了阻断，而是**换了一条没有被阻断的网络路径**。代理本身仍然遵守 TCP/IP 协议，数据包没有「消失」或「穿越」防火墙——它们走的是另一条合法的物理/逻辑链路。

### 1.3 代理系统的组成

| 组件 | 说明 | 示例 |
|------|------|------|
| 代理客户端 | 发起请求的一方，配置了代理地址 | 浏览器、curl、系统代理设置 |
| 代理服务器 | 接收并转发请求的中间服务器 | Squid、Shadowsocks、Clash、V2Ray 服务端 |
| 代理协议 | 客户端与代理服务器之间的通信协议 | HTTP CONNECT、SOCKS5、Shadowsocks、VMess |
| 目标服务器 | 客户端真正想访问的服务器 | google.com、internal-api.company.com |
| 路由规则（可选） | 决定哪些流量走代理，哪些直连 | PAC 文件、Clash 规则、Surge 规则 |

> 这五个组件中，「代理协议」决定通信格式，「路由规则」决定分流策略。两者经常被混淆：协议是「怎么说」，规则是「走哪条路」。协议分类见本篇第 3 节，规则模式见[机制篇](proxy-system-tun-dns.md)。

## 2. 按方向分类：正向代理、反向代理和透明代理

> 这是最容易混淆的一组概念。区分标准只有两个问题：**谁配置了代理？** 和 **代理为谁服务？**

三种代理的「方向」不是指数据流向（请求和响应都是双向的），而是**代理站在谁的立场上**：

```text
正向代理 — 站在客户端这边，代理「向外」访问
反向代理 — 站在服务端这边，代理「向内」接收请求
透明代理 — 站在网络管理者这边，客户端被「不知不觉」代理
```

---

### 2.1 正向代理（Forward Proxy）

**正向代理服务于客户端**，是客户端主动配置的。这是日常「科学上网」场景中接触到的代理类型。

```text
                   ┌─→ 服务器 A
用户 → 正向代理 ───┼─→ 服务器 B
                   └─→ 服务器 C
```

**工作流程：**

1. 用户在浏览器/系统中配置代理地址（如 `127.0.0.1:7890`）
2. 浏览器不直接解析目标域名，而是把 HTTP 请求发给代理
3. 代理解析目标域名，自行 DNS 查询，向目标发起新 TCP 连接
4. 目标服务器看到的来源 IP 是代理的 IP

**特征：**

- 客户端**主动配置**代理地址——所以也叫「显式代理」(Explicit Proxy)；
- 代理**位于客户端一侧**，从客户端视角是「向外」的；
- 目标服务器不知道真实客户端是谁，只知道代理 IP；
- 一台正向代理可以同时服务多个客户端，也可以同时访问多个目标服务器。

**典型用途：**

| 场景 | 说明 |
|------|------|
| 访问受限网站 | 通过位于不同网络环境的代理访问被封锁的目标 |
| 隐藏客户端 IP | 目标服务器只看到代理 IP，看不到真实客户端 |
| 企业内网出口统一 | 所有员工通过同一个代理出口访问外网，便于审计和过滤 |
| 绕过 IP 限制 | 某些服务限制单 IP 请求频率，通过多个代理分散请求 |

日常使用的 Clash、V2Ray、Shadowsocks 客户端均属于正向代理。

> **与透明代理的关系**：正向代理中的「显式代理」要求客户端主动配置；而透明代理（§2.3）也是正向代理的一种，但不要求客户端配置。两者的**服务方向相同**（都是帮客户端访问外部），但**接入方式不同**（主动配置 vs 被动截取）。

---

### 2.2 反向代理（Reverse Proxy）

**反向代理服务于目标服务器**，是服务端主动部署的。客户端不知道它的存在。

```text
                    ┌─→ 后端服务器 1
用户 → 反向代理 ───┼─→ 后端服务器 2
                    └─→ 后端服务器 3
```

**工作流程：**

1. 用户在浏览器中输入 `api.example.com`，DNS 解析到反向代理的 IP
2. 浏览器认为自己直接连接了 `api.example.com` 的服务器
3. 实际上连接的是反向代理（如 Nginx），反向代理根据负载、路由规则选择后端
4. 反向代理向后端发起新连接，获取响应后返回给用户

**特征：**

- 客户端**不知道**反向代理的存在，以为自己直接访问目标服务器；
- 代理**位于服务端一侧**，从客户端视角是「向内」的；
- 客户端不知道真实后端服务器是谁——可以随时替换后端而不影响客户端；
- 一台反向代理可以将请求**分发**到多台后端（负载均衡）。

**典型用途：**

| 场景 | 说明 |
|------|------|
| 负载均衡 | 将请求分发给多台后端，避免单点过载 |
| SSL 终结 | 反向代理处理 HTTPS 加解密，后端只需处理 HTTP，减轻计算压力 |
| 缓存静态资源 | 反向代理缓存图片、CSS、JS，直接返回而不穿透到后端 |
| 隐藏后端架构 | 外界只看到反向代理，不知道后端有几台服务器、用什么技术栈 |
| 统一入口认证 | 在反向代理层做身份验证，未认证请求不进入后端 |
| API 网关 | 反向代理 + 路由 + 限流 + 认证 = API 网关（如 Kong、APISIX） |

Nginx、Caddy、HAProxy、Traefik 最常见的用法就是作为反向代理。

> **与正向代理的对称性**：正向代理隐藏客户端，反向代理隐藏服务端——一个代理「外面不知道里面是谁」，另一个代理「里面不知道外面是谁」。如果你同时理解了这两种，代理的「中间人」本质就完全通了。

---

### 2.3 透明代理（Transparent Proxy）

透明代理是本章的重点——它经常被简单定义为「客户端不需要配置的代理」，但这个定义太浅，不足以理解它的使用场景和技术本质。

#### 什么是透明代理

**透明代理**指：客户端不主动配置代理地址，流量被网络设备或操作系统**自动转发**到代理服务器。

关键在「透明」——**对客户端而言**，代理的存在不可见；客户端以为自己直接访问目标服务器。

```text
客户端以为：  客户端 ────────────→ github.com
实际路径：    客户端 ─→ 网关/系统转发规则 ─→ 透明代理 ─→ github.com
```

客户端并没有手动填写代理地址，是网络中的某个环节把流量「截下来」交给了代理。

#### 与显式代理（普通代理）的对比

| 维度 | 显式代理（Explicit Proxy） | 透明代理（Transparent Proxy） |
|------|--------------------------|------------------------------|
| 配置方 | 客户端主动配置代理地址 | 客户端无需配置，网络/系统自动接管 |
| 客户端感知 | 知道代理存在 | 不知道代理存在 |
| 请求目标 | 请求发给代理地址（如 127.0.0.1:7890） | 请求直接发给目标服务器（如 github.com:443） |
| 流量进入方式 | 客户端主动连接代理端口 | 网络设备/系统规则拦截并重定向 |
| 典型场景 | 个人科学上网、开发调试 | 企业审计、校园网过滤、ISP 缓存 |

**生活类比：**

- **显式代理** = 你明确把包裹寄到代收点，再让代收点转寄给收件人。你主动选择了代收点。
- **透明代理** = 你按正常地址寄出包裹，但小区出口统一把所有包裹送到检查中心，检查中心处理后再继续寄出。你没有主动选择检查中心，甚至可能不知道它存在。

#### 实现方式

透明代理的核心技术问题是：**如何在客户端不知情的情况下，把它的流量引导到代理进程？**

有两种主流实现层次：

**a) 网络设备级别（路由器/网关）**

适用于局域网场景——所有设备流量经过路由器，路由器负责截取和转发。

```text
手机 ─┐
电脑 ─┤
平板 ─┼──→ 家庭/企业路由器 ─→ 透明代理 ─→ Internet
IoT  ─┘
```

实现手段：
- **策略路由（Policy Routing）**：根据源 IP、目标端口等条件，将特定流量路由到代理
- **NAT 重定向**：使用 iptables/nftables 的 REDIRECT 或 DNAT 目标，将流量改写到代理端口
- **TPROXY**：Linux 的透明代理技术，可以在不修改 IP 包的情况下将流量交给本地代理进程，同时保留原始目标地址（让代理知道客户端原本想访问谁）

**b) 操作系统级别（本机流量劫持）**

适用于单台设备——代理客户端在操作系统内设置规则，把本机产生的流量重定向到自己的代理进程。

实现手段：

| 技术 | 层级 | 说明 |
|------|------|------|
| iptables REDIRECT | 网络层 | 将本机出站流量重定向到本地代理端口，但会丢失原始目标地址 |
| iptables TPROXY | 网络层 | 重定向的同时保留原始目标地址，代理无需猜测客户端想访问谁 |
| nftables | 网络层 | iptables 的继任者，支持更灵活的规则表达 |
| 策略路由 | 网络层 | 通过路由表将流量引向特定网卡或代理进程 |
| TUN 虚拟网卡 | 网络层 | 创建虚拟网卡接管流量，由用户态程序处理，详见[机制篇](proxy-system-tun-dns.md) |

**TUN 模式就是透明代理在操作系统级别最常用的实现**——代理客户端创建虚拟网卡 → 修改路由表 → 所有出站 IP 包先到虚拟网卡 → 代理进程读取 → 分流处理。

```text
应用程序
   ↓
操作系统网络栈
   ↓
TUN 虚拟网卡
   ↓
代理客户端
   ├── 直连：原始 socket 发出
   ├── 代理：封装后发往代理服务器
   └── 拦截：丢弃（如广告域名）
```

#### 为什么需要透明代理

透明代理解决的核心问题：**有些程序不支持配置代理，或者逐台配置代价太高**。

| 场景 | 为什么需要透明代理 |
|------|--------------------|
| 游戏程序 | 不支持 HTTP/SOCKS 代理设置 |
| 命令行程序 | 很多不读 `HTTP_PROXY` 环境变量 |
| 手机/电视/IoT 设备 | 不方便逐台设置代理 |
| 局域网统一管理 | 希望所有设备通过同一出口，不需逐台配置 |
| 企业审计/过滤 | 强制所有流量经代理审查，用户无法绕过 |
| 校园/机场网络 | 强制用户通过认证页面（captive portal） |
| ISP 透明缓存 | 运营商透明代理缓存热门内容，节省出口带宽 |

#### 透明代理会修改数据吗？

透明代理只定义**流量的接入方式**，不规定**对流量的处理程度**。

- **可以不修改**：代理只做转发，不检查内容——对于 HTTPS 流量，透明代理通常只能看到目标 IP、端口和 SNI（如果未加密 SNI），无法看到请求正文。
- **也可以深层次处理**：对于 HTTP 明文流量，代理可以修改内容；对于 HTTPS，如果代理实施**中间人解密（MITM）**——让客户端信任代理签发的证书——则可以解密、检查甚至修改 HTTPS 内容。

> [!WARNING]
> 中间人解密在企业环境中常见（公司签发内部 CA 证书并推送到员工电脑），但如果在你不知情的情况下发生，这就是安全攻击。浏览器会显示证书警告。

#### 「透明」到底是对谁透明？

**「透明」是指对客户端透明**——客户端无需感知、无需配置。

但不是所有人都看不见代理：

- 客户端：不知道（如果实现得当）
- 网络管理员：知道（是他部署的）
- 代理服务器：知道（流量经过它）
- 目标服务器：**仍然看到代理出口 IP**，不知道真实客户端 IP（除非代理在 HTTP Header 中注入了 `X-Forwarded-For`）
- 高级检测：通过 TCP/IP 指纹、时序分析等手段，某些情况下可以推断出透明代理的存在

> 更准确的理解：**客户端无需感知、无需配置，但代理并非完全无法被发现。**

#### 透明代理与 TUN、规则代理、全局代理的关系

这四个概念经常被一起提及，但它们**不在同一个维度**：

| 概念 | 描述的是什么 |
|------|-------------|
| 透明代理 | **流量怎么进入代理**（客户端主动配置 vs 被动截取） |
| TUN 模式 | **透明接管的技术实现**（虚拟网卡） |
| 规则代理 | **流量进入后怎么分流**（哪些直连，哪些走代理） |
| 全局代理 | **分流策略的一种**（所有匹配流量都走代理） |

它们的关系可以这样排列：

```text
应用流量
   ↓
TUN 虚拟网卡 / 路由重定向 / iptables   ← 透明代理的实现层
   ↓
流量进入代理客户端                      ← 透明代理 vs 显式代理的分界线
   ↓
规则判断                                ← 规则代理 / 全局代理 的决定点
   ├── 国内网站：直连
   ├── 国外网站：走代理
   └── 广告域名：拦截
```

所以完全可能存在：**TUN 模式 + 透明代理 + 规则分流**，或者 **TUN 模式 + 透明代理 + 全局代理**——它们是正交维度，不是互斥选项。

---

### 2.4 三者深度对比

| 维度 | 正向代理 | 反向代理 | 透明代理 |
|------|----------|----------|----------|
| 谁配置 | 客户端 | 服务端 | 网络管理员 |
| 代理为谁服务 | 客户端 | 服务端 | 网络管理者 |
| 客户端是否感知 | 是（主动配置） | 否（以为直连） | 否（完全不知情） |
| 流量进入方式 | 客户端主动连接代理 | 客户端连接代理 IP（DNS 指向代理） | 网络层拦截并重定向 |
| TCP 连接模型 | 两段（客户端↔代理 + 代理↔目标） | 两段（客户端↔代理 + 代理↔后端） | 两段（同正向代理） |
| 隐藏对象 | 客户端 IP | 后端服务器 | 客户端 IP + 代理的存在 |
| 典型位置 | 靠近客户端 | 靠近服务端 | 网络出口/网关/本机 |
| 典型工具 | Clash, V2Ray, Shadowsocks | Nginx, Caddy, HAProxy | Squid（透明模式）, iptables + TUN |

> [!NOTE]
> 透明代理本质上是一种**特殊部署方式的正向代理**——它为客户端访问外部服务，只是客户端不需要配置。正向/反向是按**服务对象**分的；透明/显式是按**接入方式**分的。

## 3. 按协议分类：HTTP 代理与 SOCKS5 代理

> 正向代理内部最常见两种协议。它们决定的是「客户端和代理之间用什么语言沟通」，而不是代理的方向。

选择 HTTP 还是 SOCKS5，取决于你的流量类型：

- 只有 HTTP/HTTPS → HTTP 代理足够
- 有 UDP（DNS 查询、QUIC、游戏）、非 HTTP TCP（SSH、数据库）→ 需要 SOCKS5 或 TUN 模式

---

### 3.1 HTTP 代理

HTTP 代理理解 HTTP 协议，工作在 OSI 第 7 层（应用层）。它能解析 HTTP 请求头和响应头，因此可以做到 SOCKS5 做不到的事情——缓存、内容过滤、Header 注入。

**两种工作模式：**

#### 普通 HTTP 请求（明文）

代理解析 HTTP 请求，理解其中的语义：

```text
客户端 → 代理：GET http://example.com/page HTTP/1.1
                 Host: example.com

代理 → example.com：GET /page HTTP/1.1    ← 注意：代理可能修改请求
                     Host: example.com
                     X-Forwarded-For: 客户端IP  ← 代理可以注入 Header
```

代理可以：
- 检查并缓存响应（下次同样请求直接返回缓存）
- 过滤恶意内容
- 添加 `X-Forwarded-For` 告知目标真实客户端 IP
- 修改 User-Agent

#### HTTPS 请求：CONNECT 隧道

HTTPS 内容是加密的，HTTP 代理无法解析。解决方案是 `CONNECT` 方法——代理建立一个 TCP 隧道，不再解析内容，只做字节流转发：

```text
# 第一步：建立隧道
客户端 → 代理：CONNECT example.com:443 HTTP/1.1
                 Host: example.com:443

代理 → 客户端：HTTP/1.1 200 Connection Established

# 第二步：隧道建立后，客户端直接在隧道上进行 TLS 握手
客户端 ↔ 代理 ↔ example.com:443
         ↑              ↑
    这段是 CONNECT 隧道，代理只转发 TCP 字节流
    不解析 TLS 内容，看不到 HTTP 请求/响应正文
```

> CONNECT 隧道模式下的 HTTP 代理，行为和 SOCKS5 非常接近——都是「不理解上层内容的字节流转发」。区别在于：CONNECT 只能代理 TCP，且建立隧道之前需要一次 HTTP 握手。

**HTTP 代理的局限：**

- 只能代理 HTTP/HTTPS（端口 80/443 或其他自定义 HTTP 端口）
- 不代理 UDP（HTTP CONNECT 隧道只支持 TCP）
- 对非 HTTP 协议的 TCP 流量（如 SSH、数据库连接）可能行为异常

---

### 3.2 SOCKS5 代理

SOCKS5 工作在 OSI 第 5 层（会话层），**不理解上层协议**。它只做一件事：在客户端和目标之间建立 TCP 或 UDP 连接，然后双向转发字节。

> 不理解协议 = 不解析 HTTP 头，不知道这是网页还是 SSH——全部当作字节流对待。好处是通用，代价是不能做内容层面的智能处理。

#### SOCKS5 握手流程

SOCKS5 的连接建立分三个阶段：

```text
阶段 1：认证协商
客户端 → 代理：我支持 [无认证, 用户名密码, GSSAPI]，你选哪个？
代理 → 客户端：用无认证

阶段 2：连接请求
客户端 → 代理：请帮我连接到 example.com:443（TCP）
代理 → 客户端：已连接成功，分配的绑定地址是 0.0.0.0:xxxxx

阶段 3：数据转发
客户端 ↔ 代理 ↔ example.com:443 （纯 TCP/UDP 字节流转发）
```

**SOCKS5 支持的认证方式：**

| 认证方式 | 说明 |
|----------|------|
| 无认证 (0x00) | 不需要任何凭据——常见于本地代理 |
| 用户名/密码 (0x02) | 明文用户名密码 |
| GSSAPI (0x01) | Kerberos 等企业认证 |

> SOCKS4 是 SOCKS5 的前身，只支持 TCP 且不支持域名解析（只能连接 IP 地址）。SOCKS4a 补上了域名支持。SOCKS5 是目前的实际标准。

**SOCKS5 的核心优势：**

- **协议无关**：可以代理任何基于 TCP/UDP 的应用（HTTP、HTTPS、FTP、SSH、MySQL、游戏）
- **UDP 支持**：这是相比 HTTP 代理的杀手级差异——DNS 查询（UDP 53）、QUIC（UDP 443）、WebRTC、在线游戏都需要 UDP
- **默认无加密**：SOCKS5 本身只是转发协议，不提供加密。但可以配合 TLS（SOCKS5 over TLS）或作为加密隧道工具的内部协议使用

> 在 Clash / V2Ray 这类现代代理工具中，SOCKS5 只是**本地入口协议**（客户端 ↔ 本地代理进程之间的通信），**出站协议**（本地代理 ↔ 远程代理服务器）另有一套加密协议（如 Shadowsocks、VMess、Trojan）。SOCKS5 在这里的职责是：让不支持代理的应用程序也能通过标准 SOCKS5 接口接入代理系统。

---

### 3.3 HTTP 代理 vs SOCKS5 代理

| 维度 | HTTP 代理 | SOCKS5 代理 |
|------|-----------|-------------|
| OSI 层级 | 应用层（第 7 层） | 会话层（第 5 层） |
| 支持协议 | HTTP/HTTPS | 任意 TCP/UDP |
| UDP 支持 | ❌ 不支持 | ✅ 支持 |
| 内容感知 | 可解析 HTTP 语义 | 不理解上层协议 |
| HTTP 缓存/过滤 | ✅ 可以 | ❌ 不可以 |
| HTTPS 处理 | CONNECT 隧道（不解密） | 纯字节流转发 |
| 性能 | CONNECT 模式接近 SOCKS5 | 纯转发，开销低 |
| 加密 | 依赖 HTTPS（代理本身不加密） | 本身不加密，可配合 TLS |
| 认证 | 基本不内置（或通过 Proxy-Authorization Header） | 内置认证协商（无认证/用户名密码/GSSAPI） |

> [!NOTE]
> 现代代理工具（Clash、V2Ray）本地通常**同时提供** HTTP 和 SOCKS5 入口。浏览器一般用 HTTP 代理（系统代理原生支持），需要 UDP 的场景（DNS、QUIC、游戏）用 SOCKS5 或 TUN 模式。

## 7. 代理 vs VPN vs 路由 vs NAT

四个概念常被混为一谈，但它们工作在完全不同的层级。

| 概念 | 工作层级 | 粒度 | 加密 | TCP 连接 | 典型场景 |
|------|----------|------|------|----------|----------|
| 代理 | 应用/会话层 | 按应用/域名 | 取决于协议 | 两段独立连接 | 访问特定网站、应用分流 |
| VPN | 网络/链路层 | 整个设备或网段 | 是 | 隧道封装 | 远程办公、隐私保护 |
| 路由 | 网络层 | 按 IP 网段 | 否 | 不变 | 网络互联、流量工程 |
| NAT | 网络层 | 按 IP+端口 | 否 | 不变（只改包头） | 内网访问公网 |

### 7.1 代理 vs VPN

最常见混淆。核心区别：

- **代理**：按应用/域名粒度，只有使用代理的应用走代理。是「选择性地走不同的路」。
- **VPN**：连接后，整个设备的流量（或指定网段）通过 VPN 隧道。是「全接管，走同一条隧道」。

```text
代理模式：
App A → 代理 → 目标 X
App B → 直连 → 目标 Y    ← 两条不同路径可以同时存在

VPN 模式：
所有 App → 加密隧道 → VPN 服务器 → 目标  ← 所有流量走同一条路
```

> [!NOTE]
> TUN 模式让代理看起来像 VPN（系统级接管），但本质不同：TUN 模式在用户态做分流——可以一部分走代理、一部分直连；VPN 是全隧道封装。TUN 模式 = 代理的分流能力 + VPN 的系统级覆盖。

### 7.2 代理 vs NAT

NAT 和代理都涉及「用另一个 IP 访问目标」，但机制完全不同：

```text
NAT：客户端 ──── 同一条 TCP 连接 ────→ 目标
                ↑ 中间路由器只改了 IP 包头

代理：客户端 ── TCP 连接 1 ──→ 代理 ── TCP 连接 2 ──→ 目标
                ↑ 两个独立 TCP 连接！
```

> [!IMPORTANT]
> 代理会**终结 TCP 连接**并建立新连接，这意味着代理可以：修改内容、更换协议版本（如 HTTP/1.1 → HTTP/2）、做缓存。NAT 只改包头，两端仍是同一条 TCP 连接，中间设备不能改内容。

### 7.3 代理 vs 路由

- **路由**决定「数据包走哪条物理/逻辑路径」——是 IP 层的转发决策，不修改数据。
- **代理**决定「由谁代表我去请求」——是应用层的转发，建立新连接，可以修改数据。

> 路由 = 地图（告诉你去某地走哪条路），代理 = 中介（你让他替你去，他用自己的身份去）。
