# 代理（Proxy）完全指南

> 代理是一台帮你转发网络请求的中间服务器。它最核心的价值在于：让不能直接通信的两端，通过一个「中间人」完成数据交换。本文从定义、分类、工作机制到实际排错，系统梳理代理相关的核心概念。

## 1. 什么是代理

### 1.1 核心定义

**代理（Proxy）** 是一台位于客户端和目标服务器之间的中间服务器。客户端不直接连接目标服务器，而是将请求发给代理，由代理代为转发；目标服务器的响应也先返回给代理，再由代理传回客户端。

```text
客户端 ────→ 代理服务器 ────→ 目标服务器
   ↑                            │
   └──────── 响应返回 ───────────┘
```

在这个模型中：
- 客户端知道代理的存在（主动配置了代理地址）；
- 目标服务器看到的请求来源是代理的 IP，而不是客户端的真实 IP。

可以把它暂时理解为**收发室**：你不住在目标城市，但你委托当地一个收发室帮你寄信和收信。对方看到的寄件地址是收发室的地址，而不是你的真实住址。

在这个类比中：
- 你 = 客户端
- 收发室 = 代理服务器
- 收信人 = 目标服务器
- 收发室的地址 = 代理 IP

但这个类比并不完全准确，因为收发室只是「帮忙转交」，而代理还可以做缓存、过滤、修改请求内容等操作。

### 1.2 为什么代理能访问客户端无法访问的网络

这是很多人第一次接触代理时的核心困惑：**为什么同一个网站，我直接访问不了，但代理能访问？**

答案在于**网络路径不同**：

```text
你的电脑 ──×── 目标服务器        （路径被阻断）
你的电脑 ──✓── 代理服务器 ──✓── 目标服务器  （路径畅通）
```

阻断通常发生在**你到目标服务器之间的某一段路径**上，而不是发生在代理到目标服务器之间。代理服务器位于不同的网络位置（通常是不同的国家或网络运营商），它到目标服务器的路径是完全不同的。

具体原因包括：

1. **IP 层面阻断**：你的 IP 地址段被目标服务器的防火墙或中间路由器的 ACL（访问控制列表）拦截，但代理的 IP 不在拦截范围内。
2. **DNS 污染/劫持**：你本地 DNS 解析到的目标 IP 是错误的或被劫持的，而代理使用自己的 DNS，能解析到正确的 IP。
3. **DPI（深度包检测）**：你的网络出口部署了 DPI 设备，能识别并阻断特定协议的流量（如 TLS SNI 字段泄露的目标域名）。代理与客户端之间的通信通常是加密的（如加密的 SOCKS5 over TLS），DPI 无法识别内容。
4. **地理封锁**：目标服务器根据 IP 地理位置拒绝你的请求，但代理位于允许的地理区域内。

> [!IMPORTANT]
> 关键结论：代理不是「绕过」了阻断，而是**换了一条没有被阻断的网络路径**。代理本身仍然遵守 TCP/IP 协议，数据包没有「消失」或「穿越」防火墙——它们走的是另一条合法的物理/逻辑链路。

### 1.3 代理系统的组成

一个完整的代理系统包含以下部分：

| 组件 | 说明 | 示例 |
|------|------|------|
| 代理客户端 | 发起请求的一方，配置了代理地址 | 浏览器、curl、系统代理设置 |
| 代理服务器 | 接收并转发请求的中间服务器 | Squid、Shadowsocks、Clash、V2Ray 服务端 |
| 代理协议 | 客户端与代理服务器之间的通信协议 | HTTP CONNECT、SOCKS5、Shadowsocks、VMess |
| 目标服务器 | 客户端真正想访问的服务器 | google.com、internal-api.company.com |
| 路由规则（可选） | 决定哪些流量走代理，哪些直连 | PAC 文件、Clash 规则、surge 规则 |

## 2. 按方向分类：正向代理、反向代理和透明代理

这是最容易混淆的一组概念，核心区别在于**谁在使用代理**和**谁配置了代理**。

### 2.1 正向代理（Forward Proxy）

**正向代理服务于客户端**，是客户端主动配置的。

```text
                   ┌─→ 服务器 A
用户 → 正向代理 ───┼─→ 服务器 B
                   └─→ 服务器 C
```

特征：
- 客户端**主动配置**代理地址；
- 代理**位于客户端一侧**（从客户端的视角看是「向外」的）；
- 目标服务器不知道真实客户端是谁，只知道代理的 IP；
- 典型用途：访问受限网站、隐藏客户端 IP、企业内网出口统一管理。

日常使用的「科学上网」工具（Clash、V2Ray、Shadowsocks 客户端）属于正向代理。

### 2.2 反向代理（Reverse Proxy）

**反向代理服务于目标服务器**，是服务端主动部署的。

```text
                    ┌─→ 后端服务器 1
用户 → 反向代理 ───┼─→ 后端服务器 2
                    └─→ 后端服务器 3
```

特征：
- 客户端**不知道**反向代理的存在，它以为自己直接访问目标服务器；
- 代理**位于服务端一侧**（从客户端的视角看是「向内」的）；
- 客户端不知道真实后端服务器是谁；
- 典型用途：负载均衡、SSL 终结、缓存静态资源、隐藏后端架构、统一入口认证。

Nginx、Caddy、HAProxy 最常见的用法就是作为反向代理。

### 2.3 透明代理（Transparent Proxy）

**透明代理对客户端完全不可见**，不需要客户端做任何配置。

特征：
- 客户端**不需要配置**代理，甚至不知道代理存在；
- 通常部署在**网络出口处**（路由器、网关），由网络管理员配置；
- 通过拦截网络层的流量（如 iptables REDIRECT / TPROXY）将请求重定向到代理；
- 典型用途：企业网络审计、内容过滤、缓存加速、ISP 的透明缓存。

> [!NOTE]
> 「透明」指的是对客户端透明（无感知），而不是对目标服务器透明。目标服务器仍然看到代理的 IP。

### 2.4 三者对比

| 维度 | 正向代理 | 反向代理 | 透明代理 |
|------|----------|----------|----------|
| 谁配置 | 客户端 | 服务端 | 网络管理员 |
| 代理为谁服务 | 客户端 | 服务端 | 网络管理者 |
| 客户端是否感知 | 是（主动配置） | 否（以为直连） | 否（完全不知情） |
| 典型位置 | 靠近客户端 | 靠近服务端 | 网络出口/网关 |
| 典型工具 | Clash, V2Ray, Shadowsocks | Nginx, Caddy, HAProxy | Squid（透明模式）, iptables |

## 3. 按协议分类：HTTP 代理与 SOCKS5 代理

正向代理内部最常见的两种协议是 HTTP 代理和 SOCKS5 代理。

### 3.1 HTTP 代理

HTTP 代理理解 HTTP 协议，能解析 HTTP 请求头和响应头。

工作方式：
- **普通 HTTP 请求**：代理解析请求，可能修改 Header（如添加 `X-Forwarded-For`），然后转发给目标。
- **HTTPS 请求**：使用 `CONNECT` 方法建立隧道。代理不解析加密内容，只在客户端和目标服务器之间建立一个 TCP 隧道，传输加密字节流。

```text
# HTTP CONNECT 方法（用于 HTTPS）
客户端 → 代理：CONNECT example.com:443 HTTP/1.1
代理 → 客户端：HTTP/1.1 200 Connection Established
客户端 ↔ 代理 ↔ example.com:443 （加密隧道，代理看不到内容）
```

特点：
- 只能代理 HTTP/HTTPS 流量；
- 可以理解和修改 HTTP 层面的信息（缓存、过滤、Header 注入）；
- `CONNECT` 隧道模式下性能较好（代理不解析内容）；
- 大部分操作系统和浏览器原生支持。

### 3.2 SOCKS5 代理

SOCKS5 工作在**会话层**（OSI 第 5 层），不理解上层协议，只负责转发 TCP/UDP 数据包。

```text
# SOCKS5 握手流程（简化）
客户端 → 代理：支持哪些认证方式？
代理 → 客户端：无需认证
客户端 → 代理：请连接到 example.com:443
代理 → 客户端：已连接成功
客户端 ↔ 代理 ↔ example.com:443 （纯转发）
```

特点：
- **协议无关**：可以代理任何基于 TCP/UDP 的应用（HTTP、HTTPS、FTP、SSH、游戏、数据库连接等）；
- **支持 UDP**：这是相比 HTTP 代理的重要优势（HTTP CONNECT 只能代理 TCP）；
- 不解析上层协议内容，无法做 HTTP 级别的缓存或过滤；
- 默认无加密，但可以通过 SOCKS5 over TLS 或配合加密隧道工具使用。

### 3.3 对比

| 维度 | HTTP 代理 | SOCKS5 代理 |
|------|-----------|-------------|
| OSI 层级 | 应用层（第 7 层） | 会话层（第 5 层） |
| 支持协议 | HTTP/HTTPS | 任意 TCP/UDP |
| UDP 支持 | 不支持 | 支持 |
| 内容感知 | 可以解析 HTTP | 不理解上层协议 |
| HTTP 缓存/过滤 | 可以 | 不可以 |
| 性能 | CONNECT 模式下接近 SOCKS5 | 略低（无内容解析，纯转发） |
| 加密 | 依赖 HTTPS | 本身不加密 |

> [!NOTE]
> 在现代代理工具（如 Clash、V2Ray）中，本地通常同时提供 HTTP 和 SOCKS5 入口。客户端可以按需选择：浏览器通常用 HTTP 代理（系统代理），需要代理 UDP 的场景（如 DNS 查询、QUIC、游戏）用 SOCKS5 或 TUN 模式。

## 4. 系统代理的工作原理

### 4.1 什么是系统代理

**系统代理**（System Proxy）是操作系统提供的一个设置项，告诉所有「遵守系统代理设置」的应用程序：「把你的网络请求发给这个地址」。

它不是操作系统强制拦截所有流量，而是一个**约定**——应用程序可以选择遵守，也可以忽略。

```text
应用程序发起网络请求
    │
    ├─ 遵守系统代理？ ──→ 读取系统代理地址 → 发给代理
    │
    └─ 不遵守？ ──→ 直接连接目标
```

### 4.2 三种工作模式

现代代理客户端（如 Clash Verge、V2RayN、Surge）通常提供三种模式：

#### 全局模式（Global）

所有匹配到的流量都走代理。简单粗暴，但可能造成以下问题：
- 国内网站也走代理，变慢；
- 代理服务器带宽被内网流量占用；
- 部分国内服务（如银行）检测到海外 IP 会拒绝服务。

#### 规则模式（Rule）

根据预定义的规则集，**按域名或 IP 分流**：

```text
流量进入
    │
    ├─ 匹配「直连规则」（如 *.cn、内网 IP） → 直接连接
    ├─ 匹配「代理规则」（如 google.com、twitter.com） → 走代理
    ├─ 匹配「拒绝规则」（如广告域名） → 拒绝连接
    └─ 未匹配 → 按默认策略处理（通常是代理或直连）
```

这是**最推荐**的日常使用模式。核心在于规则集的质量——规则越精准，体验越好。

#### 直连模式（Direct）

所有流量都不走代理，相当于 **暂时关闭代理功能**。代理客户端仍在运行，但不介入任何流量。

### 4.3 系统代理的局限性

系统代理最大的局限是**应用层代理**，只对「愿意遵守系统代理设置」的应用生效：

- ✅ 浏览器（Chrome、Firefox、Edge）
- ✅ 大部分 HTTP 客户端（curl、wget、Python requests）
- ❌ 终端（大部分 CLI 工具不读系统代理，需要手动设环境变量）
- ❌ 很多桌面应用（不查系统代理设置）
- ❌ ping、dig、nslookup 等网络层工具（不是 HTTP 应用）
- ❌ UDP 流量（大部分系统代理实现只代理 TCP）

这就是 TUN 模式要解决的问题。

## 5. TUN 模式

### 5.1 什么是 TUN 模式

TUN 模式通过创建一个**虚拟网卡**（TUN 设备），在操作系统网络栈的更低层（网络层，OSI 第 3 层）拦截流量，实现对**所有应用程序**的代理——无论该应用是否支持代理设置。

```text
应用程序（任意）
    │
    ▼
操作系统网络栈
    │
    ▼
TUN 虚拟网卡 ←── 代理客户端处理（分流、加密、转发）
    │
    ├─ 直连流量 → 物理网卡 → 目标
    └─ 代理流量 → 代理服务器 → 目标
```

### 5.2 工作原理

TUN 是 Linux/Unix 内核提供的一种虚拟网络设备，工作在**三层（IP 层）**：

1. 代理客户端创建一个 TUN 虚拟网卡（如 `tun0`）；
2. 修改系统路由表，将默认路由指向 TUN 网卡（或使用更精细的策略路由）；
3. 所有出站 IP 数据包先到达 TUN 网卡；
4. 代理客户端从 TUN 读取原始 IP 包，在用户态判断该包应该直连还是走代理；
5. 如果需要代理，将 IP 包封装到代理协议中发送；如果直连，通过原始 socket 直接发出；
6. 返回的数据包反向处理，写回 TUN 网卡，交给应用。

```text
应用发 IP 包
    │
    ▼
路由表指向 TUN
    │
    ▼
代理客户端用户态程序
    ├── 判断分流策略
    ├── 需要代理：封装 → 代理服务器
    └── 直连：原始 socket 发出
```

### 5.3 TUN 模式 vs 系统代理

| 维度 | 系统代理 | TUN 模式 |
|------|----------|----------|
| 工作层级 | 应用层（第 7 层） | 网络层（第 3 层） |
| 生效范围 | 遵守系统代理的应用 | 所有应用（系统级） |
| TCP 支持 | ✅ | ✅ |
| UDP 支持 | 依赖实现（HTTP 代理不支持） | ✅ |
| ping / ICMP | ❌ | ✅（可代理 ICMP） |
| 终端工具 | 需手动设环境变量 | 自动生效 |
| 性能开销 | 较低 | 略高（用户态处理每个 IP 包） |
| 配置复杂度 | 低 | 中（需要管理员权限创建虚拟网卡） |

> [!WARNING]
> TUN 模式需要管理员/root 权限来创建虚拟网卡和修改路由表。在 macOS 上通常使用 Network Extension API，在 Windows 上使用 Wintun 驱动。

## 6. DNS 与代理分流

DNS 是代理配置中最容易被忽视但又最容易出问题的一环。

### 6.1 为什么 DNS 需要分流

问题场景：

```text
1. google.com 的 DNS 查询通过代理服务器发出 → 返回正确的 IP
2. baidu.com 的 DNS 查询也通过代理服务器发出 → 返回海外的 baidu CDN IP → 访问变慢
```

如果 DNS 不区分内外，会出现：
- **DNS 泄漏**：本应直连的国内域名，DNS 查询却发到了海外代理，暴露访问意图；
- **CDN 错误**：国内网站被解析到海外 CDN 节点，访问速度极慢。

### 6.2 DNS 分流策略

现代代理客户端（Clash、Surge 等）通常这样处理 DNS：

```text
DNS 查询
    │
    ├─ 域名匹配「代理规则」 → 通过代理服务器进行 DNS 查询（防 DNS 污染）
    ├─ 域名匹配「直连规则」 → 使用本地 DNS（如 223.5.5.5）查询，获得国内最优 CDN
    └─ 未匹配 → 按默认 DNS 策略（通常是「并发查询，取最快响应」或走代理 DNS）
```

更高级的实现还会：
- 使用 **Fake IP** 模式：代理客户端对 DNS 查询返回一个假的 IP（通常来自私有地址段），然后在代理客户端内部维护「假 IP → 真实域名」的映射。当后续流量到达代理客户端时，通过假 IP 反查出真实域名，再进行规则匹配。这解决了 IP 层面规则匹配困难的问题（因为很多服务使用 CDN，IP 难以一眼判断归属）。
- **并发查询**：同时向本地 DNS 和代理 DNS 发起查询，取最先返回的结果。

## 7. 代理 vs VPN vs 路由 vs NAT

这四个概念经常被混淆，但它们工作在完全不同的层级和场景。

| 概念 | 工作层级 | 粒度 | 是否加密 | 双向性 | 典型场景 |
|------|----------|------|----------|--------|----------|
| 代理（Proxy） | 应用层/会话层 | 按应用/按域名 | 取决于协议 | 单向转发 | 访问特定网站、应用分流 |
| VPN | 网络层/数据链路层 | 整个设备或网段 | 是 | 双向互联 | 远程办公、隐私保护、跨区域组网 |
| 路由（Route） | 网络层 | 按 IP 网段 | 否 | 双向 | 网络互联、流量工程 |
| NAT | 网络层 | 按 IP + 端口 | 否 | 单向翻译 | 内网访问公网、端口映射 |

### 7.1 代理 vs VPN

这是最常见的混淆点。核心区别：

- **代理**：按应用/域名粒度，只有配置了代理的应用才走代理，其他流量不受影响。代理是「选择性地转发」。
- **VPN**：一旦连接，整个设备的所有流量（或指定网段）都走 VPN 隧道。VPN 是「全接管」。

```text
代理模式：
App A → 代理 → 目标 X
App B → 直连 → 目标 Y    ← 两条路径可以同时存在

VPN 模式：
所有 App → VPN 隧道 → VPN 服务器 → 目标  ← 所有流量走同一条路
```

> [!NOTE]
> 现代「代理客户端 + TUN 模式」在效果上非常接近 VPN（系统级接管所有流量），但底层机制不同：TUN 模式仍然在做代理协议的分流转发，而 VPN 做的是 IP 层隧道封装。

### 7.2 代理 vs NAT

NAT 和代理都涉及「用另一个 IP 访问目标」，但机制完全不同：

- **NAT**：工作在网络层，只修改 IP 包的源地址/目标地址，不解析上层协议。NAT 是路由器/网关的**无状态或状态化地址翻译**。
- **代理**：工作在应用层或会话层，建立了从客户端到代理再到目标的两段**独立 TCP 连接**。

```text
NAT：客户端 ──────── TCP 连接 ────────→ 目标（中间只改了 IP 包头）
代理：客户端 ── TCP 连接 1 ──→ 代理 ── TCP 连接 2 ──→ 目标（两段独立连接）
```

> [!IMPORTANT]
> 代理会**终结 TCP 连接**并建立新连接，这意味着代理可以修改内容、更换协议版本、做缓存。NAT 只是修改包头，两端仍然是同一个 TCP 连接。

### 7.3 代理 vs 路由

路由决定**数据包走哪条路**，代理决定**由谁转发**。

- 路由是 IP 层的转发决策，配置在路由器或操作系统路由表中，不对数据做任何修改。
- 代理是应用层的转发，由特定的代理软件处理，可以修改、过滤、缓存数据。

简单说：路由是一张「地图」（走哪条路），代理是一个「中介」（帮你办事的人）。

## 8. Windows、WSL 与 Docker 中的代理

这三个环境的代理配置各有难点，因为它们涉及不同的网络栈和隔离边界。

### 8.1 Windows 代理配置

Windows 的系统代理设置在「设置 → 网络和 Internet → 代理」中，本质是写入了注册表。支持：
- 手动设置 HTTP/HTTPS 代理地址和端口
- 自动配置脚本（PAC 文件 URL）
- 自动检测设置（WPAD）

常见代理客户端（Clash Verge、V2RayN）会自动修改系统代理设置，开启时设为 `127.0.0.1:7890`，关闭时还原。

**命令行代理**：大部分 Windows 命令行工具不读系统代理，需手动设置环境变量：

```powershell
set HTTP_PROXY=http://127.0.0.1:7890
set HTTPS_PROXY=http://127.0.0.1:7890
```

### 8.2 WSL 代理配置

WSL（Windows Subsystem for Linux）的网络栈是虚拟化的，它有一个独立的网络命名空间，因此：

- WSL 内部的 `localhost` 不是 Windows 的 `localhost`；
- Windows 上运行的代理客户端（监听 `127.0.0.1:7890`）在 WSL 中无法通过 `127.0.0.1` 访问。

**配置方法**：

WSL 2 中，Windows 宿主机的 IP 可以从 `/etc/resolv.conf` 中的 nameserver 获取（WSL 用它做 DNS 转发，实际指向 Windows 主机）：

```bash
# 获取 Windows 宿主 IP
export HOST_IP=$(cat /etc/resolv.conf | grep nameserver | awk '{print $2}')

# 设置代理
export HTTP_PROXY="http://${HOST_IP}:7890"
export HTTPS_PROXY="http://${HOST_IP}:7890"
export ALL_PROXY="socks5://${HOST_IP}:7890"  # 如果代理客户端提供 SOCKS5
```

可以写入 `~/.bashrc` 或 `~/.zshrc` 中持久化。

> [!WARNING]
> WSL 1 和 WSL 2 的网络架构不同。WSL 1 共享 Windows 的网络栈，可以直接用 `localhost`；WSL 2 使用独立的虚拟网络，需要通过宿主 IP 访问。以上配置适用于 WSL 2。

**TUN 模式在 WSL 中的替代方案**：如果 Windows 端代理客户端开启了 TUN 模式且允许局域网连接，WSL 可以直接将流量路由到宿主机的 TUN 网卡。但这属于高级配置，大多数情况下设环境变量即可。

### 8.3 Docker 代理配置

Docker 中的代理涉及三个层面：

#### a) Docker 守护进程（拉取镜像时使用）

Docker daemon 本身需要代理来拉取镜像（如从 `docker.io`）。配置方式：为 Docker daemon 创建 systemd override 文件或 `daemon.json`。

```json
// /etc/docker/daemon.json
{
  "proxies": {
    "http-proxy": "http://host:7890",
    "https-proxy": "http://host:7890",
    "no-proxy": "localhost,127.0.0.1,*.local"
  }
}
```

#### b) 构建镜像时的代理（`docker build`）

构建过程中 `RUN` 指令的网络请求需要代理时，使用 `--build-arg` 或 Docker 19.03+ 的自动注入：

```bash
docker build --build-arg HTTP_PROXY=http://host:7890 .
```

或在 Dockerfile 中：

```dockerfile
ARG HTTP_PROXY
ARG HTTPS_PROXY
```

#### c) 容器运行时的代理

容器内部的网络请求需要代理时，通过环境变量传入：

```bash
docker run -e HTTP_PROXY=http://host:7890 -e HTTPS_PROXY=http://host:7890 myimage
```

> [!NOTE]
> 容器内访问宿主机的代理服务时，`localhost` 指向容器自身。在 Linux Docker 上，可以用 `host.docker.internal`（Docker 20.04+）或宿主机的 Docker 网桥 IP（通常是 `172.17.0.1`）。

## 9. 常见故障与排查方法

### 9.1 排查框架

遇到代理问题时，按以下顺序排查——从最底层（网络可达性）到最上层（应用配置）：

```text
1. 代理服务器是否可达？       ← ping / telnet / nc 测试连通性
2. 代理端口是否正确？         ← 检查客户端和代理配置
3. 代理协议是否匹配？         ← HTTP 代理 vs SOCKS5，端口是否对应
4. DNS 是否正常？            ← nslookup / dig 检查解析
5. 分流规则是否正确？         ← 当前域名是否命中了预期的规则
6. 应用是否使用代理？         ← 是否设了系统代理/环境变量
7. 证书/TLS 问题？           ← 是否被中间人证书拦截
```

### 9.2 常见问题速查

#### 问题 1：代理客户端显示「已连接」，但浏览器无法访问任何网站

可能原因：
- 浏览器没有使用系统代理（检查浏览器自身的代理设置）；
- 系统代理地址或端口设置错误（浏览器代理设置指向了错误端口）；
- 代理客户端未正确修改系统代理（手动检查系统代理设置）；
- 代理客户端开启了「直连模式」。

排查方法：
```bash
# 检查代理端口是否在监听
netstat -an | grep 7890    # Linux/macOS
netstat -ano | findstr 7890  # Windows

# 用 curl 直接测试代理
curl -x http://127.0.0.1:7890 https://www.google.com -v
```

#### 问题 2：部分网站能访问，部分不能

可能原因：
- 规则模式下，某些域名的规则配置有误（该走代理的走了直连，或被拒绝）；
- DNS 分流问题：直连域名的 DNS 被污染，解析到错误 IP；
- CDN 问题：代理 IP 被目标网站的 CDN 或防火墙封锁。

排查方法：
- 检查代理客户端日志，确认当前域名的匹配规则和最终动作；
- 临时切换到全局模式测试，确认是否为规则问题；
- 检查代理客户端的 DNS 设置（是否开启了 DNS 分流）。

#### 问题 3：终端（CLI）工具不走代理

原因：大部分 CLI 工具不读取系统代理设置。

解决方法：
```bash
# Linux/macOS
export HTTP_PROXY=http://127.0.0.1:7890
export HTTPS_PROXY=http://127.0.0.1:7890
export ALL_PROXY=socks5://127.0.0.1:7890  # 如果使用 SOCKS5

# 测试
curl https://www.google.com
```

#### 问题 4：开启 TUN 模式后网络完全断开

可能原因：
- TUN 虚拟网卡路由配置错误，导致所有流量（包括到代理服务器的流量）都进了 TUN，形成环路；
- TUN 网卡与物理网卡的路由优先级冲突。

排查方法：
- 关闭 TUN 模式，切换到系统代理模式；
- 检查路由表：`ip route`（Linux）/ `route print`（Windows）/ `netstat -rn`（macOS）；
- 确保到代理服务器 IP 的路由不经过 TUN 网卡。

#### 问题 5：DNS 泄漏

现象：在代理环境下访问 `ipleak.net` 或 `dnsleaktest.com`，显示的是本地运营商的 DNS 服务器。

原因：浏览器的 DNS 查询没有走代理。

解决方法：
- 在代理客户端中开启「DNS 通过代理」或类似选项；
- 浏览器层面启用「DNS over HTTPS」（DoH），并设置为代理友好的 DoH 服务；
- TUN 模式下确认 DNS 请求也被 TUN 网卡捕获。

#### 问题 6：WSL/Docker 无法连接代理

核心问题：`127.0.0.1` 指的不是 Windows 宿主机。

排查方法：
```bash
# 在 WSL 内获取宿主机 IP
cat /etc/resolv.conf | grep nameserver

# 测试宿主机代理端口是否可达
nc -zv <宿主机IP> 7890

# 在 Docker 容器内
docker run --rm alpine nc -zv host.docker.internal 7890
```

## 常见误区

1. **「代理 = VPN」**：代理是按应用/域名粒度的选择性转发，VPN 是整个设备的网络层接管。两者机制不同，虽然 TUN 模式模糊了这条边界。

2. **「SOCKS5 比 HTTP 代理快」**：快慢取决于具体实现和网络状况，不是协议本身决定的。SOCKS5 的优势在于协议无关和 UDP 支持，不是速度。

3. **「设置了系统代理，所有流量都会走代理」**：系统代理只是一个约定，只有遵守这个约定的应用才会走代理。很多应用和命令行工具完全忽略系统代理。

4. **「代理能保护隐私」**：代理只隐藏你的 IP，不加密流量内容（除非代理协议本身包含加密层，如 Shadowsocks、VMess）。HTTP 代理对目标服务器和代理服务器之间的内容可以是明文的。

5. **「关闭代理客户端就等于停止代理」**：关闭客户端后，系统代理设置可能没有恢复，导致浏览器仍然尝试连接已经不存在的代理端口，表现为「断网」。

6. **「正向代理和反向代理是对立的」**：同一软件（如 Nginx）可以同时做正向代理和反向代理，取决于配置方式。正向/反向描述的是**角色**，不是软件类别。

## 小结

1. 代理的核心价值是**提供一条不同的网络路径**——它不能穿透防火墙，只是走了一条没有被阻断的路。
2. 正向代理服务于客户端，反向代理服务于服务端，透明代理对客户端不可见。区分它们的关键是「谁配置的」和「为谁服务」。
3. HTTP 代理工作在应用层，只能代理 HTTP/HTTPS；SOCKS5 工作在会话层，可以代理任意 TCP/UDP。
4. 系统代理是应用层的约定，TUN 模式是网络层的虚拟网卡接管，后者覆盖范围更广但配置更复杂。
5. DNS 分流是代理体验的关键——DNS 泄漏和 CDN 错误是常见的「能用但不好用」的根源。
6. 排查代理问题从底层到上层：连通性 → 端口 → 协议 → DNS → 规则 → 应用配置。
